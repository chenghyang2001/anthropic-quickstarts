<project_specification>
  <project_name>CodeReviewBot — AI-Powered Code Review Tool</project_name>

  <overview>
    CodeReviewBot is an AI-powered code review tool that combines a CLI interface with an optional
    web dashboard. Developers run "codereview scan ./src" locally or point it at a GitHub PR, and
    Claude returns structured findings (critical / warning / suggestion) with file-line context.
    All findings are persisted in SQLite so teams can track quality trends over time.

    Primary audience: Individual developers and small engineering teams wanting automated
    pre-merge code review without sending code to a third-party SaaS.
  </overview>

  <core_features>
    <local_file_scanner>
      - Recursively scan a directory for Python, TypeScript, and Go source files
      - Respect .codereviewbotignore patterns (same syntax as .gitignore)
      - Chunk large files into overlapping segments (max 200 lines, 20-line overlap) before sending to Claude
      - Display real-time progress bar (tqdm) as files are processed
      - CLI command: codereview scan ./src --config .codereviewbot.yaml
    </local_file_scanner>

    <github_pr_integration>
      - Authenticate via GITHUB_TOKEN environment variable
      - Fetch PR diff using PyGithub, reconstruct per-file diffs
      - Post review findings as inline PR comments via GitHub Review API
      - Support --dry-run flag to preview comments without posting
      - CLI command: codereview pr --repo owner/repo --pr 42
    </github_pr_integration>

    <severity_classification>
      - Claude returns structured JSON: findings array with severity, line, message, suggestion fields
      - Three severity levels: critical (blocks merge), warning (should fix), suggestion (optional)
      - CLI exit code 1 if any critical findings exist (suitable for CI gates)
      - Color-coded terminal output using Rich library
    </severity_classification>

    <review_history_storage>
      - Every scan result saved to SQLite with timestamp and repo context
      - Query history: codereview history --repo myrepo --last 30d
      - Diff two reviews to see quality improvement over time
      - Pruning command: codereview history prune --older-than 90d
    </review_history_storage>

    <streamlit_dashboard>
      - Summary cards: total findings by severity this week
      - Line chart: findings over time (critical trend) via Plotly
      - Table: top 10 files with most findings in current week
      - Team statistics: findings per author (from git blame data)
      - Filter by repo, date range, severity, author
    </streamlit_dashboard>

    <configurable_rules>
      - .codereviewbot.yaml at project root defines focus areas
      - Enable/disable rule categories: security, performance, style, logic
      - Custom prompt additions: "Also check for use of deprecated requests patterns"
      - Per-language settings (Go-specific idioms, Python type-hint enforcement)
    </configurable_rules>

    <export_reporting>
      - codereview report --format markdown — full Markdown report to stdout
      - codereview report --format github-comment — post summary comment to PR
      - Report includes: file list, finding count by severity, top issues, remediation hints
      - Scheduled reports via cron: weekly digest email via smtplib
    </export_reporting>
  </core_features>

  <database_schema>
    <tables>
      <reviews>
        - id INTEGER PRIMARY KEY AUTOINCREMENT
        - repo_name TEXT NOT NULL
        - branch TEXT
        - pr_number INTEGER
        - scan_type TEXT NOT NULL (local | github_pr)
        - started_at DATETIME NOT NULL
        - finished_at DATETIME
        - total_files INTEGER DEFAULT 0
        - created_by TEXT (git user or system)
      </reviews>

      <findings>
        - id INTEGER PRIMARY KEY AUTOINCREMENT
        - review_id INTEGER NOT NULL REFERENCES reviews(id)
        - file_path TEXT NOT NULL
        - line_start INTEGER
        - line_end INTEGER
        - severity TEXT NOT NULL (critical | warning | suggestion)
        - category TEXT (security | performance | style | logic)
        - message TEXT NOT NULL
        - suggestion TEXT
        - suppressed BOOLEAN DEFAULT 0 (user marked as false-positive)
      </findings>

      <repo_configs>
        - id INTEGER PRIMARY KEY AUTOINCREMENT
        - repo_name TEXT UNIQUE NOT NULL
        - config_yaml TEXT (serialized YAML content)
        - updated_at DATETIME NOT NULL
      </repo_configs>

      <team_stats>
        - id INTEGER PRIMARY KEY AUTOINCREMENT
        - review_id INTEGER NOT NULL REFERENCES reviews(id)
        - author TEXT NOT NULL
        - file_path TEXT NOT NULL
        - critical_count INTEGER DEFAULT 0
        - warning_count INTEGER DEFAULT 0
        - suggestion_count INTEGER DEFAULT 0
      </team_stats>
    </tables>
  </database_schema>

  <key_interactions>
    <local_scan_cli>
      1. Developer runs: codereview scan ./src --config .codereviewbot.yaml
      2. CLI discovers all .py, .ts, .go files respecting .codereviewbotignore
      3. ReviewEngine splits each file into max 200-line chunks with 20-line overlap
      4. For each chunk, Claude API called with system prompt + code + focus rules from config
      5. Claude returns JSON: {"findings": [{"severity", "line", "message", "suggestion", "category"}]}
      6. Pydantic validates JSON response before DB write
      7. Findings deduplicated (same file + line + message) and persisted to SQLite
      8. Rich table printed to terminal with color-coded severity
      9. Exit with code 1 if any critical findings found, else exit 0
    </local_scan_cli>

    <github_pr_review>
      1. Developer runs: codereview pr --repo owner/repo --pr 42
      2. PyGithub authenticates with GITHUB_TOKEN, fetches PR diff
      3. Per-file diffs reconstructed from unified diff format
      4. Same ReviewEngine processes each changed file chunk
      5. If --dry-run: print findings to terminal without posting
      6. If not dry-run: post findings as inline review comments via GitHub Review API
      7. Add summary comment to PR with count by severity (critical/warning/suggestion)
      8. Exit code 1 if critical findings found
    </github_pr_review>

    <streamlit_dashboard_navigation>
      1. User opens http://localhost:8501
      2. Streamlit sidebar: select repo from dropdown, pick date range
      3. Dashboard tab: summary cards (st.metric) + Plotly critical trend chart
      4. Files tab: top 10 offending files table, click row to expand finding details
      5. Team tab: per-author Plotly bar chart, sortable author table
      6. Findings can be marked as suppressed (false-positive) via toggle in detail view
      7. Suppressed findings excluded from counts in all charts and tables
    </streamlit_dashboard_navigation>
  </key_interactions>

        - Call Anthropic SDK: claude-sonnet-4-6, structured JSON output, timeout=30s
        - Pydantic schema: FindingSchema with severity, line_start, line_end, message, suggestion, category
